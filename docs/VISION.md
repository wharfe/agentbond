# agentbond — 将来展望メモ

status: 方向性メモ・仮説（実装仕様ではない）
updated: 2026-03

---

## 出発点にある直感

人間コミュニティでスケールしなかったトークンエコノミーが、
AIエージェントという参加者を得て成立しやすくなる可能性がある。

人間向けのトークン設計が機能しにくかった理由は技術ではなく、
継続参加コスト・熱量の波・プロデューサー依存という構造的な問題だった。
エージェントはルール追従・反復実行・微小インセンティブ処理に向いており、
一部のトークン的設計は人間市場より agent 市場のほうが成立しやすい条件を持つ。

これは「人間では失敗、AIなら成功」という断定ではなく、
**成立条件が違う** という観察に近い。

## agentbondが押さえるレイヤー

```
今作っているもの（MVP）
├── AuthorizationToken  権限台帳（誰が何をどこまでできるか）
├── BudgetLedger        消費台帳（いくら使ったか）
└── AuditRecord         行為履歴（何をなぜやったか）

将来的に加わるもの
├── Contract            債務関係（何を約束したか）
└── Settlement          清算履歴（何を支払ったか）
```

これはエージェント経済圏における**法務・会計・統制の基盤データ**になりうる。
信用も、保証も、保険も、すべてこの観測可能なデータの上に積み重なる。

agentbondを「認可ミドルウェア」としてだけでなく、
**責任帰属を計算可能にするインフラ**として位置づけられる可能性がある。

## 経済圏の成長ロードマップ（仮説）

人間の金融市場が数百年かけて発展した道筋を、
エージェント経済圏はかなり速いペースで辿る可能性がある。

```
Phase 1: 権限・記録
  AuthorizationToken + AuditRecord の蓄積
  → 「誰が何をしたか」が観測可能になる

Phase 2: 観測・分類
  Performance Metrics（成功率・超過率・deny内訳・人手介入率）
  Incident Taxonomy（損害化した事故の種類・原因分類）
  → 「どれだけ信頼できるか」が計測可能になる

Phase 3: 信用
  @agentbond/credit — 信用スコア・トラックレコード
  → 信用があれば前払い不要になる

Phase 4: 保証・エスクロー        ← 保険より先に来る可能性
  成果未達時の自動返金
  検収通過まで支払保留
  失敗時の一部補填プール
  → contract + settlement + audit で自然に実装できる

Phase 5: 与信・貸借
  将来の成果物・収益を担保にした信用枠
  → credit line・先払い委託

Phase 6: 保険・デリバティブ      ← 特に注目
  判断責任の分界と補償
  リスクの第三者移転
  → 人間レイヤーへの露出が起きやすい

Phase 7: 所有権・持分（遠景）
  DAO的な共同所有・収益分配
  → 現時点では抽象度が高く、時期は不明
```

## 保険への注目

金融商品の中で保険だけが持つ特徴として、
損失が発生したとき最終的な引き受け手が必要になる点がある。
エージェント経済圏が十分に大きくなると、その「引き受け手」が
人間・企業レイヤーに露出する場面が出てくる可能性がある。

### 保険設計の核心：判断責任の所在

高額な実務においては、以下の順で重要度が上がる傾向がある。

```
支払えるか
指示どおり動いたか
誰の責任か          ← ここが保険設計の核心
事故時に誰が穴を埋めるか
```

責任分界点が法的・技術的に曖昧なうちは保険市場が立ち上がりにくい。
**AuditRecord + IntentRecord で判断の経緯が追跡可能になると、
責任分界点が明確になり、保険が設計しやすくなる可能性がある。**

### agentbondのデータが保険産業に持つ意味

AuditRecordの蓄積は保険会社にとってのアクチュアリアルデータになりうる。

「このタイプのエージェントがこのスコープで動いたとき、
 どのくらいの頻度でBUDGET_EXCEEDEDやSCOPE_MISMATCHが起きるか」

この統計が取れると、そのまま損害率の推計に使える可能性がある。
人間の保険が何十年もの事故データを使って料率を設計するのと同じことが、
エージェント経済圏では最初から構造的に可能かもしれない。

### 近未来の具体的シナリオ

企業がエージェントに経営判断の一部を委任しはじめると、
「エージェント役員賠償責任保険（AI D&O保険）」のような商品が出てくる可能性がある。
D&O保険は既存の保険会社が得意とする領域なので、既存プレイヤーが参入しやすい。

**テレマティクス保険との類比**
テレマティクス保険が運転データを使って保険料を個別化したように、
「agentbondのAuditRecordを提出すると保険料が下がる」という仕組みが生まれると、
インフラとしての採用動機が強くなる可能性がある。

**価値訴求の方向性**
「agentbondを入れると安全になる」より
「agentbondを入れると保険料や統制コストが下がる」
のほうが企業に刺さりやすい。

```
better governance
better trust
lower risk cost
lower underwriting friction
```

## 人間の金融と何が違うか

### 速度
清算サイクルが日単位→ミリ秒単位へ。
金融商品の時間軸が根本的に短くなる可能性がある。

### 透明性
判断の経緯が追跡できる。説明責任を切り出せる。
オンチェーンの実績があれば信用評価がアルゴリズム的に透明になりうる。

### 能力変動リスク（人間にはない新しいリスク）
エージェントは「夜逃げ」しないが、以下のような制度・基盤側の変化で
突然性能が変わるリスクがある。

```
モデル更新・バージョン変更
API仕様変更
ツール接続の変化
コスト変動・レート制限
safety policy変更
プロバイダー依存
```

将来の信用スコアは単純な一元値ではなく、

```
reliability            成功率・安定性
capability stability   能力の変動幅
policy volatility      運用ポリシーの変化リスク
provider dependency    特定基盤への依存度
```

のような分解が必要になるかもしれない。

## 将来のモジュール構想

```
@agentbond/core           型定義（今作っている）
@agentbond/auth           認可エンジン（MVP）
@agentbond/intent         意図証明
@agentbond/contract       エージェント間契約
@agentbond/settlement     マイクロペイメント
                             ↓ ここから先が将来展望
@agentbond/metrics        性能指標・観測データ
@agentbond/incident       事故分類・インシデント記録
@agentbond/credit         信用スコア・トラックレコード
@agentbond/guarantee      保証・エスクロー
@agentbond/audit-export   保険会社・監査向けデータフィード
@agentbond/insurance      保険契約のスマートコントラクト連携
```

`@agentbond/audit-export` は保険だけでなく、
SOC/ISO監査・内部統制・規制当局向け提出・事故調査・外部審査にも広がりうる。

AuditRecordのスキーマを今から意識して設計しておく価値がある。

## 暗号資産・ブロックチェーンとの接点

エージェント間の決済に既存の金融インフラは相性が悪い場合がある。
（銀行口座を持てない・マイクロペイメントのコストが高い）

ブロックチェーンネイティブなウォレットとトークンが活きやすい条件：
- 特定の中央管理者を置きたくない
- クロスプラットフォームで動く
- 人間の監督なしに清算したい

これらが揃ったとき、暗号資産が経済圏の決済レイヤーになりうる。
agentbondのSettlementLayerはその接続口として設計されている。
